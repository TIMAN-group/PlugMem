# PlugMem CLI & Daemon — Design

Status: draft, pre-implementation
Owner: jizej
Last updated: 2026-05-03

## Goal

Reduce the friction of running PlugMem from "clone a repo, set six env
vars, launch uvicorn manually" to a single `plugmem` CLI with an
OpenClaw-style setup wizard. After installation, any user should be able
to run `plugmem init && plugmem start` and have a working memory backend
without reading source.

## Non-goals

- **Hosted PlugMem service.** No managed `plugmem.ai`. Everything is
  self-hosted, like OpenClaw itself.
- **Bundling LLM weights.** PlugMem requires an LLM and an embedding
  endpoint; the CLI doesn't ship them. The wizard helps the user wire
  up their own (cloud API key or local Ollama / vLLM / LM Studio).
- **Multi-tenancy / auth.** The existing `PLUGMEM_API_KEY` mechanism is
  enough for single-user self-host. Auth UX is unchanged.
- **Replacing the FastAPI app.** The CLI is a thin Python wrapper around
  `plugmem.api.app:app`; no backend logic moves into the CLI.

## Framing

PlugMem today is a research codebase that happens to expose a service.
The set of steps to actually run it — clone, install Python deps, write
out env vars (LLM_BASE_URL, LLM_API_KEY, LLM_MODEL, EMBEDDING_BASE_URL,
EMBEDDING_MODEL, PLUGMEM_API_KEY), `mkdir` data dirs, run uvicorn — is
itself a friction wall higher than most users will climb.

OpenClaw solved this with a setup wizard that asks a few targeted
questions and writes a config. We mirror that shape:

- One install command (`uv tool install plugmem` or `pipx install plugmem`).
- One init command (`plugmem init`) that prompts for what's actually
  needed.
- One start command (`plugmem start`) that backgrounds the service.

The wizard's hardest job: it must validate every choice (LLM
connectivity, embedding connectivity, port availability, data-dir
writeability) *before* writing config, so first-time users don't end up
with a broken install they have to debug.

## Architecture

### CLI layer

- **Framework:** Typer (Click underneath; nice defaults for help text +
  prompt UX). Add Rich for the wizard's formatted output.
- **Distribution:** new `[project.scripts] plugmem = "plugmem.cli:app"`
  entry in the existing `pyproject.toml`. No new package; the CLI lives
  in `plugmem/cli/` alongside the existing API code so they share the
  same dependency surface and version.
- **Install paths:** `uv tool install plugmem` (preferred since the repo
  already uses uv) or `pipx install plugmem`. Both isolate the CLI from
  the user's project venvs.

### Config

- **Location:** `~/.config/plugmem/config.toml` (XDG; `$XDG_CONFIG_HOME`
  honored).
- **Format:** TOML with sections matching the existing env-var families
  one-to-one. Env vars override config (standard 12-factor).

```toml
[service]
port = 8080
host = "127.0.0.1"
api_key = "<auto-generated on init>"
data_dir = "~/.local/share/plugmem/chroma"
log_level = "INFO"

[llm]
provider = "openai"      # openai | azure | ollama | vllm | other
base_url = "https://api.openai.com/v1"
api_key = "..."
model = "gpt-4o-mini"

[embedding]
provider = "ollama"       # openai | azure | ollama | other
base_url = "http://127.0.0.1:11434/v1"
api_key = ""
model = "nomic-embed-text"
```

The CLI translates this into the env vars the FastAPI app already
reads (`LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL`,
`EMBEDDING_BASE_URL`, `EMBEDDING_MODEL`, `PLUGMEM_API_KEY`,
`CHROMA_PATH`, etc.) before exec'ing uvicorn. **No FastAPI code
changes** — just env-var translation.

### Daemon

- **Library:** `python-daemon` for the daemon-mode fork (PID file,
  stdio detachment, signal handling). DIY fork is too easy to get
  wrong.
- **State paths:**
  - PID: `~/.local/state/plugmem/plugmem.pid`
  - Log: `~/.local/state/plugmem/plugmem.log` (rotated via
    `logging.handlers.RotatingFileHandler` — 10 MB × 5)
  - Data: `~/.local/share/plugmem/chroma/` (configurable; symlink-friendly)
- **Foreground mode:** `plugmem start --foreground` skips the fork and
  streams logs to stdout — easier debugging, useful for `tmux` /
  `screen` users who'd rather manage the process themselves.
- **Auto-start at login (optional):** `plugmem install-launchd` on
  macOS / `plugmem install-systemd` on Linux generates the appropriate
  unit file in user scope. Not run automatically — user opts in.

### LLM / embedding endpoints

- **BYO only.** The wizard never installs models or starts a model
  server. It detects, validates, and configures.
- **Ollama auto-detect:** at wizard start, probe
  `http://127.0.0.1:11434/api/tags`. If reachable, list pulled models
  and offer them as defaults for both LLM and embedding sections. This
  is the single largest UX win for hobbyists who already run Ollama —
  removes "what URL?" and "what model name?" prompts entirely.
- **Validation:** for OpenAI-compatible endpoints, send a minimal
  completion request with the chosen model. For embeddings, a one-token
  embedding call. Loop on failure with edit/retry options.

## Subcommands

| Command | Behavior |
|---|---|
| `plugmem init` | Interactive wizard. Writes config; refuses to overwrite without `--force`. |
| `plugmem start [--foreground]` | Launch uvicorn. Daemon by default; PID + log under state dir. Idempotent (no-op if already running). |
| `plugmem stop` | SIGTERM the daemon by PID; SIGKILL after timeout. |
| `plugmem restart` | `stop` then `start`. |
| `plugmem status` | Show running/stopped, PID, port, data dir, last health probe. |
| `plugmem logs [-f]` | Tail the daemon log; `-f` to follow. |
| `plugmem health` | One-shot `/health` request; exit non-zero if any `*_available` flag is false. |
| `plugmem config edit` | Open `$EDITOR` on the config file; re-validate on save. |
| `plugmem config get <key>` | Print one config value (e.g. `llm.model`). |
| `plugmem config set <key> <value>` | Set one config value. Triggers validation for the touched section. |
| `plugmem doctor` | Run all wizard checks against existing config; print a checklist (no prompts). |
| `plugmem graphs ls / stats <id> / rm <id>` | REST-API wrappers for poking at memory state without curl. |
| `plugmem install-launchd` / `install-systemd` | Emit and install user-scope auto-start unit. |

## Init wizard flow

Sequential prompts; each section validates before moving on. Every step
has a `[edit]` / `[retry]` / `[skip]` option on validation failure.

1. **LLM endpoint**
   - Detect Ollama on `127.0.0.1:11434`. If found: list models, offer
     one as default.
   - Otherwise prompt for provider type (openai / azure / vllm / other),
     then `base_url`, `api_key` (skip for local-only providers),
     `model`.
   - Validate by sending a 1-token completion request.

2. **Embedding endpoint**
   - Same shape as LLM. If Ollama detected and `nomic-embed-text` (or
     similar) is pulled, offer it. Otherwise prompt.
   - Validate by sending a 1-input embed request.

3. **Service settings**
   - Port (default 8080; check it's free, suggest +1 if taken).
   - Data dir (default `~/.local/share/plugmem/chroma`; create + verify
     writable).
   - Generate `service.api_key` automatically (32-byte token, hex). Show
     it once; user can override if they have an existing one.
   - Log level (default INFO; debug as a checkbox).

4. **Final probe**
   - Start the service in foreground briefly (10s).
   - Hit `/health` — confirm `llm_available`, `embedding_available`,
     `chroma_available` all `true`.
   - On success: stop, write config, print a summary including a
     copy-pastable env-var block for wiring up the Claude Code plugin
     (`export PLUGMEM_BASE_URL=... PLUGMEM_API_KEY=...`).
   - On failure: keep config in memory, offer to re-run failed sections.

5. **Optional**: prompt to start the service now (`plugmem start`) and
   to install user-scope auto-start.

## Implementation plan

Sequenced like the coding-agent stages: ship small, verify, expand.

### Stage CLI-1 — Scaffold + init wizard + start/stop (3–5 days)

- Add `plugmem/cli/` with Typer app skeleton.
- Add `[project.scripts] plugmem = "plugmem.cli:app"` to `pyproject.toml`.
- Implement config TOML read/write.
- Implement `init` wizard (LLM, embedding, service sections + final
  probe).
- Implement `start` (daemon + foreground), `stop`, `status`, `logs`,
  `health`.
- Pytest coverage for: config round-trip, wizard branch coverage with
  mocked HTTP, start/stop/status state machine.
- Verify end-to-end on a clean macOS user: `uv tool install plugmem &&
  plugmem init && plugmem start && plugmem health`.

### Stage CLI-2 — Doctor, graphs, polish (2–3 days)

- `plugmem doctor` runs the wizard's validators against existing
  config; emits a checklist.
- `plugmem graphs ls / stats / rm` (REST wrappers).
- `plugmem config edit / get / set` (granular edit).
- `plugmem install-launchd` / `install-systemd` for auto-start.
- README and ONBOARDING update: "Quick start" becomes the CLI flow.

### Stage CLI-3 — Distribution (1–2 days)

- Verify `uv tool install` and `pipx install` both work.
- Tag a release; confirm GitHub release artifacts are buildable.
- Optional: Docker image (`Dockerfile` + GHCR publish) for users who
  don't want Python on their box. Same config TOML, same env-var
  translation.

## Open questions

1. **Config schema migration.** When fields change (e.g., adding a new
   provider type), how do existing configs upgrade? Versioned schema +
   `config.version` field; CLI reads and migrates on `init`/`start`
   if behind. Defer until first breaking change.

2. **Multi-instance support.** Some users may want two PlugMem
   instances on the same machine (e.g., one for personal, one for
   work). Current design is single-instance. Could support
   `plugmem --profile <name>` reading
   `~/.config/plugmem/profiles/<name>.toml`. Not blocking; add when
   asked.

3. **Wizard idempotency.** `plugmem init --force` overwrites; what
   about `plugmem init --reconfigure llm` (just the LLM section)?
   Useful UX; defer to CLI-2.

4. **Auto-update.** Should the CLI check for new versions and prompt
   the user, OpenClaw-style? Probably yes (one-line check via PyPI),
   but defer until distribution is solid.

5. **Telemetry.** Do we collect any anonymized usage telemetry to
   guide eval / dev priorities? Default off; opt-in switch at init.
   Aligned with self-host philosophy.
